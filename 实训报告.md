## 二、实习主要内容

### 2.1 项目背景与业务场景

OM-Agent（绿盟 IDS/IPS 自主运维智能体）是一个面向绿盟科技入侵检测/防御系统（IDS/IPS）网络安全设备的自动化运维系统。传统运维模式下，当设备出现故障（如 Web 管理界面无法访问、引擎不转发流量、规则不生效等），需要资深运维工程师通过 SSH 登录设备，逐层排查 Web 层、Python 管理层、Server 引擎层、Class 检测引擎层及系统资源层，过程繁琐、依赖个人经验且耗时长。

本项目将《系统后台排查手册》（百余页运维文档）中的排查方法转化为 107 个可执行诊断技能，结合 DeepSeek V4 大语言模型实现两种自动化运维工作流：**针对性故障排查**和**全链路架构巡检**。

### 2.2 技术选型与架构设计

经过对 Agent 领域主流技术方案的调研与对比，最终确定以下技术栈：

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| LLM 编排 | LangChain + LangGraph | LangGraph 提供 DAG 状态图引擎，支持复杂的条件路由与多轮迭代推理，天然适配诊断排查的逐步深入场景 |
| 大模型 | DeepSeek V4 Flash | 高性价比、支持 OpenAI 兼容 API、中文理解能力强 |
| SSH 传输 | AsyncSSH | 异步非阻塞，支持多设备并发连接与保活 |
| Web 后端 | FastAPI + WebSocket | 高性能异步框架，原生 WebSocket 支持，满足实时流式推送需求 |
| 前端 | Vue 3 + Element Plus | 轻量级 SPA，CDN 引入无需构建工具，快速交付 |
| 数据库 | SQLite + SQLAlchemy 2.0 | 轻量级嵌入式数据库，无需额外部署，适合运维工具场景 |
| 密码加密 | Fernet (AES-128-CBC + HMAC) | 工业级对称加密，保护 SSH 设备密码 |
| CLI | Click | Python 主流的命令行框架，支持 CLI 和 Web 两种使用方式 |

项目采用**分层模块化架构**：

```
Vue 3 SPA (前端交互层)
    ↕ REST / WebSocket
FastAPI 服务层 (API 网关)
    ↕
LangGraph 工作流引擎 (编排核心)
    ↕
Skill Registry (107 个诊断技能，4 个层级)
    ↕
AsyncSSH 传输层 (远程设备连接)
    ↕
NSFOCUS IDS/IPS 目标设备
```

### 2.3 核心模块实现

#### 2.3.1 SSH 传输层（`src/transport/ssh_client.py`）

封装 AsyncSSH 实现异步 SSH 连接管理，核心设计要点：

- 异步连接池管理，支持命令执行超时控制（默认 30s）
- 命令结果封装为 `SSHResult` 数据类（含退出码、stdout、stderr、执行耗时）
- SSH 协议级保活（TCP/SSH keepalive 心跳包）
- 支持 `known_hosts` 主机密钥验证
- 原始字节返回（encoding=None），避免二进制输出破坏连接

#### 2.3.2 技能层（`src/skills/`）

将《系统后台排查手册》中所有排查手段转化为 107 个程序化 Python 函数，按系统架构分为 5 个层级：

| 层级 | 技能数 | 典型检查项 |
|------|:-----:|-----------|
| Web 层 | 20 | Nginx 状态、PHP-FPM 进程、端口监听、错误日志、License、Redis、PostgreSQL、SSL 证书 |
| Python 管理层 | 18 | daemon 日志、guard 看门狗、共享内存 GG 标记、ZMQ 端口、webtoid 事件桥接 |
| Server 引擎层 | 22 | 引擎心跳、bypass 标记、DPDK 大页内存、网卡链路、coredump、XML 校验 |
| Class 引擎层 | 18 | 实例输出、IPS 规则、Hyperscan、tcmalloc 堆指标、协议解码器 |
| 系统资源层 | 15 | CPU、内存、磁盘、进程、僵尸进程、dmesg 错误、文件描述符、DNS |
| 通用 | 14 | `exec` 万能命令、文件检查、进程检查、日志读取等基础工具 |

技能系统采用**自动注册机制**：`SkillRegistry` 单例通过反射自动发现所有 `check_*` 函数，维护技能元数据（名称、描述、分类、参数签名），支持批量执行和按层级检索。

#### 2.3.3 LangGraph 工作流引擎（`src/graph/engine.py`）

这是项目的核心编排引擎，共 2256 行代码，实现两个独立工作流：

**工作流 A：针对性故障排查（三节点动态分析链）**

```
用户输入故障 + 可选截图文件
    → plan_node: LLM 分析故障症状 + 系统架构上下文，生成跨 4 层覆盖的初始诊断计划（4-6 个技能）
    → execute_node: 依次执行计划中的诊断技能，通过 WebSocket 实时推送进度
    → evidence_node: LLM 评估执行结果，判断哪些假设被证实/排除，识别关键未知
    → deep_plan_node: 从仅未执行的技能中选择 1-2 个深挖方向，内置 25 个关键词映射
    → validate_node: 5 种程序化检测（bypass/coredump/OOM/GG 标记/磁盘），零 LLM 调用
    → decide_v2: 路由决策 → 继续执行 / 全局重评估(replan) / 生成报告
    → report_node: 输出结构化 Markdown 诊断报告
```

**工作流 B：全链路架构巡检（深度扫描）**

```
connect → full_link_web → full_link_python → full_link_engine → full_link_system → full_link_report
```

逐层执行所有检查项，对每个异常项触发 LLM 驱动的三段式分析（现象描述 → 深挖排查链 → 综合诊断结论），最终生成包含设备健康评分（A-F 级）的完整报告。

**引擎的关键设计亮点：**

- **三节点排查链拆分**：将原来的 459 行单体分析节点重构为 `evidence → deep_plan → validate`，各司其职，代码可维护性大幅提升
- **技能去重机制**：deep_plan 节点只展示未执行技能，从根本上杜绝 LLM 推荐重复技能
- **证据→技能自动映射**：25 个关键词规则（如 "oom" → check_oom_logs + check_memory + check_dmesg_errors），程序化补充 LLM 可能遗漏的检查方向
- **技能质量分级**（H/M/L）：39 个关键技能按诊断价值分级，deep_plan 优先推荐高价值技能
- **replan 限流**：仅在 iter=2,4 时触发全局重评估（最多 2 次），避免无限循环

#### 2.3.4 大模型集成与 Prompt 工程

采用 LangChain 的 `ChatOpenAI` 兼容接口连接 DeepSeek V4，关键配置：

- `temperature=0`（诊断场景需确定性输出，避免幻觉）
- `max_tokens=4096`
- 超时 120s
- JSON 输出格式自动清理（`_sanitize_llm_json`），处理 LLM 输出中的常见格式错误（如未转义反斜杠）

在 Prompt 工程方面：
- 将系统架构摘要（四层结构、通信方式、启动依赖链）直接注入 LLM 的 System Prompt
- 提供"正常状态速查表"，防止 LLM 将正常现象误判为异常（如 DPDK server 进程 800% CPU 占用为其正常轮询行为）
- 设计分步骤的输出格式约束，确保 LLM 返回结构化的、可解析的诊断计划

#### 2.3.5 FastAPI 后端服务（`src/api/server.py`）

实现 20+ 个 REST API 端点，覆盖：
- 设备 CRUD 管理
- 两种工作流的触发与状态查询
- 历史记录的分页查询与筛选
- SSH 保活管理
- WebSocket 实时流推送（`/ws/stream/{session_id}`）

安全措施：
- API Key 认证中间件（常量时间比较防止时序攻击）
- 文件上传校验（单文件 ≤ 10MB，总计 ≤ 50MB，最多 10 个文件）
- 错误信息脱敏
- Session ID 使用完整 UUID v4（122 位熵）

#### 2.3.6 Vue 3 前端 Dashboard（`static/`）

采用 CDN 方式引入 Vue 3、Element Plus、Axios 和 marked，无需构建步骤即可运行。包含四个功能标签页：

| 标签页 | 功能 |
|--------|------|
| 设备管理 | 设备 CRUD，SSH 密码加密保存 |
| 执行任务 | 选择设备和工作流 → 启动，实时展示进度、层状态卡片、终端日志和 Markdown 报告 |
| 历史记录 | 分页列表，按类型/状态筛选 |
| 报告详情 | Markdown 渲染，现象 + 诊断结论 + 命令输出 + 深挖排查链 |

#### 2.3.7 安全加固

完成了全面的安全审查与修复，共发现 18 个安全隐患（5 个严重、3 个高危、7 个中危），已完成 15 个修复：

- SSH 密码使用 Fernet 加密存储，启动时自动迁移明文密码
- 所有 API 端点强制认证
- 默认绑定 127.0.0.1，不暴露公网
- `.gitignore` 排除 `.env`、`*.db`、`venv/`
- API Key 不再硬编码在源码中

## 三、实习总结

### 3.1 技术收获

1. **Agent 系统的完整认知**：从零构建了一个生产就绪的 LLM Agent 系统，深入理解了 LangChain 的模型抽象、LangGraph 的状态机编排、工具/Skill 注册模式、以及 Prompt 工程在 Agent 可靠性中的关键作用。认识到"LLM + 程序化检测"的混合架构比纯 LLM 方案更可靠、更高效。

2. **异步编程实践**：全栈异步（AsyncSSH → LangGraph → FastAPI → WebSocket）的设计带来了显著的性能优势，也让我深刻理解了 `async/await` 编程模型中的并发控制、超时处理和异常传播。

3. **安全工程意识**：通过安全审查实践，建立了"安全左移"的意识——密码加密、API 认证、输入校验、错误脱敏、凭据隔离等措施应在设计阶段就纳入考虑，而非事后补救。

4. **全栈开发能力**：项目覆盖了从 SSH 协议层到 Vue 3 前端的所有技术栈层级，锻炼了全局架构思维和跨层问题排查能力。

### 3.2 工程能力提升

1. **需求分析与架构设计**：从百余页运维手册中提取核心逻辑，设计出 107 个技能的系统化知识表示方案，体会到了"好的架构设计是成功的一半"。

2. **迭代开发与持续优化**：6 轮测试迭代的过程让我认识到，软件质量不是一次性写出来的，而是在反复测试、发现问题、针对性优化中逐步提升的。每一轮迭代都有明确的改进目标和可量化的效果评估。

3. **代码质量意识**：通过类型提示（Type Hints）、Pydantic 数据校验、完善的异常处理、详细的文档注释，保证了 2000+ 行核心引擎代码的可维护性。

4. **团队协作与沟通**：在跨角色团队中学会了从不同视角理解需求——产品经理关心用户体验、测试关心边界覆盖、技术支持关心可运维性，这些视角的融合让产品更加完善。

### 3.3 个人体会

本次实训最深刻的体会是：**企业级软件开发和学校课程项目有本质区别**。

在学校的课程设计中，程序跑通、功能正确就是满分；但在企业实训中，代码只是起点，真正考量的是：

- **可靠性**：SSH 断开如何处理？LLM 返回格式异常如何容错？超时如何优雅降级？
- **安全性**：密码如何存储？API 如何认证？错误信息会不会泄露内部细节？
- **可运维性**：日志是否清晰？出问题能否快速定位？配置是否灵活？
- **用户体验**：进度是否实时可见？报告是否易读？操作是否直观？

这些维度的考量让软件从一个"能用的程序"变成了"可信赖的产品"。实训还让我认识到，AI Agent 并非万能，在诊断系统的设计中，"LLM 推理 + 程序化检测"的混合策略才是工程上的最优解——让 LLM 做它擅长的高层推理和语言理解，让确定性代码做它擅长的精确校验和事务处理，二者互补成就了一个真正可用的智能运维系统。

最后，感谢实训期间导师和团队成员的指导与协作，这段经历将是我从学校走向职场的宝贵桥梁。
