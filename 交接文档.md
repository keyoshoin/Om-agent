## 1. 项目概述

### 1.1 项目定位

Om-Agent 是面向 NSFOCUS IDS/IPS 设备**开发阶段的故障定位辅助工具**。在设备开发和问题复现阶段，通过 SSH 连接远程设备，结合大语言模型自动执行诊断计划、分析输出、定位根因，生成结构化报告，帮助研发人员快速定位问题、缩短排障周期。

> **核心定位:** 开发调试阶段的故障排查与根因分析。可兼用于日常巡检，但不替代正式运维系统。

### 1.2 核心功能

| 功能 | 说明 | 交互方式 |
|------|------|----------|
| 针对性故障排查 (Workflow A) | 根据故障现象输入，LLM 自动规划诊断步骤，逐层深挖定位根因 | Web / CLI / API |
| 全链路架构巡检 (Workflow B) | 对设备四大组件 107 个检查项逐个扫描，异常自动触发深挖排查 | Web / CLI / API |
| 设备管理 | 管理 SSH 连接信息，密码加密存储，支持保活连接 | Web / API |
| 运行历史 | 排查/巡检记录持久化，按类型/状态/设备筛选，报告回放 | Web / API |
| 实时进度推送 | WebSocket 推送排查实时进度、命令输出、阶段状态 | Web |

### 1.3 技术栈概览

| 层级 | 技术选型 | 用途 |
|------|---------|------|
| LLM | DeepSeek V4 (OpenAI 兼容协议) | 故障分析、诊断规划、证据评估 |
| 工作流引擎 | LangGraph (StateGraph) | 状态机编排、节点路由 |
| Web 框架 | FastAPI + Uvicorn | REST API + WebSocket |
| 前端 | Vue 3 + Element Plus (CDN) | SPA Dashboard |
| 数据库 | SQLite + SQLAlchemy (async) | 设备信息 + 运行记录 |
| SSH | AsyncSSH | 异步远程连接 |
| 加密 | Fernet (AES-128-CBC + HMAC) | 密码加密存储 |
| CLI | Click | 命令行工具 |

## 4. 核心模块详解

### 4.1 LangGraph 引擎 (`src/graph/engine.py`)

**文件规模:** ~2256 行，是项目最核心的模块。

**职责:** 定义两个 LangGraph 工作流，编排 LLM 调用、SSH 命令执行、结果分析、路由决策的全流程。

**关键节点:**

| 节点 | 工作流 | 职责 | LLM 调用 |
|------|:------:|------|:--------:|
| `connect_node` | A + B | 建立 SSH 连接，记录远程工作目录 | - |
| `plan_node` | A | LLM 分析故障现象，从注册表选 4~6 个优先技能 | ✅ |
| `execute_skill_node` | A | 从技能队列取出一个执行（支持 `exec:cmd` 直接命令） | - |
| `evidence_node` | A | LLM 证据评估 — 判断异常、支撑/排除假设、识别未知 | ✅ |
| `deep_plan_node` | A | LLM 从仅未执行技能中选择深挖方向 + 程序化注入 | ✅ |
| `validate_node` | A | 纯程序化校验 — bypass/coredump/OOM/GG/Disk 检测 | - |
| `replan_node` | A | 周期性全局重评估 (iter=2,4 触发，最多 2 次) | ✅ |
| `report_node` | A | 生成排查报告（排查链表格 + 诊断结论 + 完整输出） | ✅ |
| `decide_v2` | A | 路由决策 — execute / replan / report | - |
| `_run_layer_check` | B | 通用层检查执行器（含 SSH 断连自动重连） | - |
| `full_link_report_node` | B | 汇总四层结果，LLM 综合生成诊断结论 | ✅ |

**关键数据结构:**

```python
# 技能质量分级 (P1-1)
SKILL_QUALITY: dict[str, dict[str, str]]
# 诊断价值: high (18个) / medium (15个) / low (6个)

# 关键词强制注入表 (兜底机制)
_KEYWORD_MANDATORY_SKILLS: list[tuple[list[str], list[str]]]
# 13 组关键词 → 必备技能映射

# 证据 → 技能自动映射
auto_inject_map: dict[str, list[str]]
# 25 个关键词 → 针对性技能注入

# 证据链 (P1-2)
Hypothesis: id, description, layer, status, supporting_evidence, refuting_evidence
EvidenceGraph: hypotheses[], cross_checks[]
```

**LLM 调用优化策略:**

| 策略 | 说明 |
|------|------|
| prompt 分阶段 | evidence / deep_plan 两个独立 prompt，避免单次过长 |
| 仅未执行技能 | deep_plan 只展示未执行技能列表，从根源消除重复建议 |
| 正常状态速查 | prompt 中内置正常状态清单，避免 DPDK 轮询误判为异常 |
| 根因判定铁律 | 状态证据优先于日志、至少 2 条独立证据、具体可修复 |
| 反验证机制 | 判定根因时要求 LLM 同时给出证伪方法 |
| JSON 容错 | `_sanitize_llm_json()` 修复 LLM 输出的非法转义字符 |

### 4.2 技能系统 (`src/skills/`)

**架构:** 注册表模式 — 技能函数按层分布在 4 个模块中，通过 `SkillRegistry` 统一注册、查找、执行。

```
SkillRegistry (registry.py)
  ├── register_all_skills()  # 自动扫描 4 个模块注册所有 check_* 函数
  ├── get(name)              # 按名查找
  ├── execute(name, client)  # 按名执行
  ├── list_all()             # 列出全部
  ├── format_skill_table()   # 格式化供 LLM prompt 使用
  └── count                  # 技能总数
```

**技能分布:**

| 模块 | 层级 | 数量 | 职责范围 |
|------|------|:----:|---------|
| `web_layer.py` | Web 管理界面 | 20 | Nginx/PHP-FPM/Redis/PG/端口/日志/License/SSL |
| `python_layer.py` | Python 管理脚本 | 18 | daemon/guard/ZMQ/webtoid/scheduler/共享内存 |
| `engine_layer.py` | 引擎层 | 40 | Server数通/Class检测/bypass/XML/coredump/DPDK |
| `sys_resource.py` | 系统资源 | 15 | CPU/内存/磁盘/网络/进程/Dmesg/OOM/FD |
| (exec 万能命令) | 通用 | 14 | 自定义命令/文件检查/日志读取 |

**SkillResult 数据模型 (`base.py`):**

```python
@dataclass
class SkillResult:
    skill_name: str       # 技能标识名
    command: str          # 实际执行的命令
    status: str           # ok | warning | error
    summary: str          # 人类可读的诊断摘要
    raw_result: SSHResult # 原始 SSH 执行结果
    parsed: dict          # 解析后的结构化数据
```

### 4.3 SSH 传输层 (`src/transport/ssh_client.py`)

**核心类:** `SSHClient` — 封装 AsyncSSH，提供连接管理和命令执行。

```python
class SSHClient:
    async def connect(host, port, username, password)  # 建立连接
    async def execute(command, timeout) -> SSHResult   # 执行命令
    async def disconnect()                              # 断开连接
    @property is_connected -> bool                       # 连接状态

@dataclass
class SSHResult:
    command: str       # 执行的命令
    stdout: str        # 标准输出
    stderr: str        # 标准错误
    exit_code: int     # 退出码 (-1 表示异常)
    error: str         # 异常信息
    timed_out: bool    # 是否超时
    duration_ms: int   # 执行耗时
```

**超时策略:**

| 类型 | 默认值 | 配置项 |
|------|--------|--------|
| 连接超时 | 15s | `SSH_CONNECT_TIMEOUT` |
| 普通命令 | 30s | `SSH_COMMAND_TIMEOUT` |
| 耗时命令 | 60s | `SSH_LONG_COMMAND_TIMEOUT` |

### 4.4 FastAPI 服务层 (`src/api/server.py`)

**路由分组:**

| 路由组 | 方法 | 说明 |
|--------|------|------|
| `/api/devices/*` | CRUD | 设备管理（密码加密/脱敏） |
| `/api/troubleshoot` | POST | 针对性排查（支持 multipart 文件上传） |
| `/api/inspect` | POST | 全链路巡检 |
| `/api/status/{id}` | GET | 查询执行状态 |
| `/api/report/{id}` | GET | 获取 Markdown 报告 |
| `/api/history/*` | CRUD | 运行历史（分页 + 筛选） |
| `/api/keepalive/*` | POST/GET | SSH 保活管理 |
| `/ws/stream/{id}` | WebSocket | 实时进度推送 |
| `/` | GET | 重定向到前端 |
| `/static/*` | - | 静态文件托管 |

**中间件:**

```
auth_middleware  →  所有 /api/* 和 /ws/* 路径验证 X-API-Key
```

**安全措施:**

- API 层异常通过 `_sanitize_error()` 脱敏后返回
- 密码获取使用 POST 避免 URL 泄露
- Session ID 使用完整 UUID v4 (122 位熵)
- 文件上传限制: 单文件 ≤10MB, 总量 ≤50MB, 最多 10 个

### 4.5 前端 SPA (`static/`)

**技术方案:** Vue 3 CDN + Element Plus CDN + Axios + marked.js

**四个 Tab 页:**

| Tab | 组件 | 功能 |
|-----|------|------|
| 设备管理 | `DevicesPage` | CRUD 设备，保存/编辑 SSH 连接信息 |
| 执行任务 | `ExecutePage` | 选择设备 → 选择工作流 → 输入故障描述 → 启动；实时进度卡片 + 终端日志 + 报告 |
| 历史记录 | `HistoryPage` | 分页列表，按类型/状态/设备筛选，自动刷新 |
| 报告详情 | `ReportDetail` | Markdown 渲染（marked.js），折叠式排查链展示 |

**认证流程:**

```
首次访问 → 弹出 API Key 输入弹窗
         → 存入 sessionStorage (仅当前标签页)
         → Axios 拦截器自动附加 X-API-Key 请求头
         → 401 响应自动清除缓存 + 重新弹出弹窗
```

### 4.6 密码加密模块 (`src/crypto.py`)

**算法:** Fernet (AES-128-CBC + HMAC-SHA256)

**存储格式:** `enc:<base64_ciphertext>`

**迁移机制:** 启动时数据库初始化过程中检测未加密密码（不以 `enc:` 开头），自动加密并回写（幂等操作）。

### 4.7 保活管理器 (`src/keepalive_manager.py`)

**功能:** 为每个设备维护独立的 SSH 后台心跳任务。

| 特性 | 说明 |
|------|------|
| 独立心跳 | 每个设备一个异步任务，定时发送 echo 命令 |
| 自动重连 | 检测到断连后自动尝试重连 |
| 智能暂停 | 巡检/排查期间自动暂停保活，避免 SSH 连接冲突 |
| 自动恢复 | 巡检/排查结束后自动恢复保活 |
| 密码同步 | 设备密码更新后自动同步到运行中的保活任务 |
| 状态查询 | 提供状态 API：连接状态、心跳次数、最后心跳时间 |

### 5.2 内存数据

**sessions (会话存储):**

```python
sessions: dict[str, dict]       # session_id → {host, username, status, ...}
```

## 6. API 接口清单

### 6.1 认证方式

所有 `/api/*` 路由需要在请求头中携带 `X-API-Key`。WebSocket 通过查询参数 `?token=`

### 6.2 设备管理

```
GET    /api/devices              设备列表（密码脱敏为 ***）
POST   /api/devices              添加设备（密码自动加密）
PUT    /api/devices/{id}         编辑设备
DELETE /api/devices/{id}         删除设备 + 停止保活
POST   /api/devices/{id}/password  获取解密密码
```

### 6.3 工作流执行

```
POST /api/troubleshoot          针对性排查（multipart/form-data，支持文件上传）
    参数: host, port, username, password, error_input, max_iterations
          device_id, session_id, files[]

POST /api/inspect               全链路巡检（application/json）
    参数: host, port, username, password, device_id, session_id

GET  /api/status/{session_id}   查询执行状态
GET  /api/report/{session_id}   获取 Markdown 报告
GET  /api/sessions              列出所有活跃会话
```

### 6.4 保活管理

```
POST /api/keepalive/start            启动保活
POST /api/keepalive/stop/{device_id} 停止保活
GET  /api/keepalive/status/{device_id}  查询状态
GET  /api/keepalive/list              列出全部
```

### 6.5 运行历史

```
GET    /api/history       历史列表（分页: page, page_size；筛选: workflow_type, device_id, status）
GET    /api/history/{id}  运行详情（含完整报告）
DELETE /api/history/{id}  删除记录
```

### 6.6 WebSocket

```
WS /ws/stream/{session_id}?token=<API_KEY>
```

**推送事件类型:**

| 事件类型 | 说明 | 关键字段 |
|---------|------|---------|
| `ssh_connected` | SSH 连接结果 | host, status, error |

| `workflow_start` | 工作流开始 | workflow_type |
| `layer_start` | 层检查开始 | layer, total |
| `cmd_start` | 命令开始执行 | command |
| `cmd_done` | 命令执行完成 | command, exit_code, stdout_preview, duration_ms |
| `layer_done` | 层检查完成 | layer, passed, total, warnings, errors |
| `llm_planning` | LLM 规划中 | phase |
| `llm_analyzing` | LLM 分析中 | phase |
| `llm_replanning` | LLM 重规划中 | phase |
| `workflow_complete` | 工作流完成 | findings_count / layer_results |

## 11. 已知问题与限制

### 11.1 架构限制

| 问题 | 影响 | 缓解措施 |
|------|------|---------|
| **内存会话存储** | 服务重启后活跃 session 丢失 | 运行记录已持久化到 SQLite；重启后需重新发起任务 |
| **SQLite 单写** | 高并发时写入瓶颈 | 当前为本地开发工具，并发量低，暂不构成问题 |
| **单 SSH 连接** | 不支持同时操作多台设备 | 设计如此，一个 session 对应一台设备 |
| **CDN 前端依赖** | 离线环境无法使用 Web Dashboard | CLI 模式不依赖 CDN，可正常使用 |
| **LLM 响应不确定性** | 相同输入可能返回不同技能选择 | temperature=0.0 降低不确定性；关键词强制注入兜底 |

### 11.2 功能限制

| 限制 | 说明 |
|------|------|
| **设备型号耦合** | 107 个技能的检查命令依赖 NSFOCUS IDS/IPS V5.6 的特定路径和进程名 |
| **LLM 依赖** | 排查质量依赖 LLM 能力，模型降级或不可用时效果显著下降 |
| **单用户设计** | 前端为单用户模式，无用户管理和权限分级 |
| **无操作审计** | 仅记录任务执行结果，不记录谁在何时执行了什么操作 |
| **报告无持久化导出** | 报告只能通过 Web 查看或 CLI `--output` 保存，无 PDF/HTML 导出 |
| **SSH 密钥认证** | 当前仅支持密码认证，不支持 SSH 密钥对 |

### 11.3 性能数据

| 指标 | 典型值 | 说明 |
|------|--------|------|
| SSH 连接建立 | 2~5s | 取决于网络延迟 |
| 单个技能执行 | 1~30s | 简单检查 1s，大日志读取 30s |
| LLM 单次调用 | 5~30s | 取决于 prompt 长度和模型负载 |
| 针对性排查 (完整) | 3~15 min | 取决于迭代次数和故障复杂度 |
| 全链路巡检 (完整) | 5~20 min | 取决于设备状态和异常数量 |
| 内存占用 | 200~500 MB | Python 进程 + LLM SDK |
