### 1.2 启动依赖链

```
guard.py (看门狗)
  └─ daemon.py (核心守护进程)
       ├─ guardProc.py → daemon.d/*.run 进程 (通用进程)
       │    ├─ nginx, php-fpm, redis, sshd
       │    ├─ guard.py, disk_monitor.py, log_agent
       │    ├─ fsm.py, oam.py, report.py, rlmd.py
       │    └─ ha.py, com_service.py, conf_mgt 等
       │
       ├─ guardServer.py → swbypass (软件bypass) → 必须先启动
       │                → server (数通引擎)      → nice -n -20
       │
       ├─ guardClass.py → class (检测引擎) → 需server存活≥15秒
       │                                     → nice -n -10
       ├─ guardLicense.py (授权监控)
       ├─ guardClock.py (NTP时钟同步)
       ├─ guardTlsProxy.py (TLS代理)
       └─ guardDisk.py (磁盘监控)
```

### 1.3 进程间通信方式

| 通信方式 | 用途 | 详情 |
|----------|------|------|
| DPDK Ring Buffer | Server ↔ Class 数据包传递 | 共享内存三环缓冲 (triring) |
| ZeroMQ (ZMQ) | 配置下发 / 服务调用 | daemon:62000, webii:62015, oam:62010 等 |
| 共享内存 (/var/daemon_info) | 守护进程控制 | "G" 字节 = 正常运行模式 |
| Unix 信号 | 引擎控制 | SIGUSR1=重载配置, 38=时间同步, 44=停止 |
| 文件状态 | 心跳 / 同步 | /tmp/server_stat, /tmp/fw_rule/* |
| TCP/UDP (webtoid) | 日志/事件传输 | 端口 50002 |
| HTTP (Flask) | 刀片间通信 | com_service 端口 8090 |

## 2. Web 管理界面排查

### 2.1 Web 组件工作原理

```
浏览器 HTTPS :443
    │
    ▼
Nginx (反向代理 + TLS终止)
    │  FastCGI → 127.0.0.1:9000
    ▼
PHP-FPM (entry.php)
    │  rest_startup() → 验证URL/认证/授权
    ▼
Dispatch → dispatch/{模块}/resourse.php
    │  ZMQ/HTTP/Shell 调用后端
    ▼
Python 管理层 → 数据库 / 引擎
```

**请求生命周期：**
1. Nginx 接收 HTTPS 请求（443端口）
2. `/api` 路径通过 FastCGI 转发给 PHP-FPM（127.0.0.1:9000）
3. `entry.php` 入口 → `rest_startup()` 初始化
4. 读取时区配置 (`/opt/nsfocus/etc/timezone.conf`)
5. 验证 License (`/opt/nsfocus/etc/eoi.agent.lic`)
6. URL 验证 + API签名认证（SHA-256 HMAC）
7. 角色权限检查
8. 路由到对应的 `dispatch/{模块}/resourse.php`
9. 执行业务逻辑（查DB/调ZMQ/调Shell）
10. 返回 JSON 响应

### 2.2 Web 常见问题排查

#### 问题现象1: 页面打不开 / 返回 502

**可能原因：** Nginx 未运行 / PHP-FPM 挂死 / 端口被占用

**排查步骤：**
```bash
# 1. 检查 Nginx 进程
ps aux | grep nginx

# 2. 检查 PHP-FPM 进程
ps aux | grep php-fpm

# 3. 检查端口监听
netstat -ntlp | grep -E '443|9000'

# 4. 检查 Nginx 错误日志
tail -100 /var/log/nginx/error_log

# 5. 检查 PHP 错误日志
tail -100 /var/log/php/error.log

# 6. 重启 Web 服务 (通过API或命令行)
/bin/super killall nginx    # 杀掉nginx，daemon会自动拉起
killall -9 php-fpm           # 杀掉php-fpm，daemon会自动拉起
```

#### 问题现象2: 登录失败 / API 返回 401

**可能原因：** 时间不同步 / API签名错误 / 用户被禁用 / License过期

**排查步骤：**
```bash
# 1. 检查系统时间
date
hwclock

# 2. 检查 NTP 同步状态
ntpdate -q <NTP_SERVER>   # 查询NTP服务器

# 3. 检查 PHP 错误日志
tail -100 /var/log/php/error.log | grep -i "auth\|login\|401"

# 4. 检查 License 状态
cat /opt/nsfocus/etc/eoi.agent.lic
# 或通过 API: GET /system/license/info

# 5. 检查 Redis (会话存储)
redis-cli ping
redis-cli keys "*" | head -20
```

#### 问题现象3: API 返回 503 / 500

**可能原因：** PHP执行出错 / 后端ZMQ服务不可达 / 数据库异常

**排查步骤：**
```bash
# 1. 查看 PHP 错误日志(最重要)
tail -200 /var/log/php/error.log

# 2. 查看 Nginx 访问日志，确认请求是否到达
tail -50 /var/log/nginx/access_log_web

# 3. 检查 PostgreSQL 数据库状态
ps aux | grep postgres
su - postgres -c "psql -c 'SELECT 1;'"  # 测试连接

# 4. 检查 ZMQ 服务是否可达
# 查看服务端口监听
netstat -ntlp | grep -E '62000|62015|62010'

# 5. 检查磁盘空间(空间满会导致日志写失败)
df -h
```

#### 问题现象4: Web 界面响应缓慢

**排查步骤：**
```bash
# 1. 检查 PHP-FPM 进程数是否耗尽
ps aux | grep php-fpm | wc -l

# 2. 检查 Nginx 连接数
ss -ant | grep :443 | wc -l

# 3. 检查内存使用
free -h

# 4. 检查数据库连接数
su - postgres -c "psql -c 'SELECT count(*) FROM pg_stat_activity;'"

# 5. 检查磁盘 IO
iostat -x 1 3
```

## 3. Python 管理脚本排查

### 3.1 核心守护进程工作原理

#### guard.py (看门狗)
- **作用：** 外部看门狗，每3秒检查 daemon.py 是否存活
- **逻辑：** 如果 daemon.py 不在运行 → `nice -n -10 daemon.py` 重启
- **启动方式：** 由 daemon.py 自身的 `check_alive()` 方法在检测到自己僵死时启动
- **进程名：** `/opt/nsfocus/bin/guard.py`

#### daemon.py (核心守护进程)
- **作用：** 整个系统的进程管理器，使用 APScheduler 调度所有守护任务
- **两个调度器：**
  - `blockSched` (BlockingScheduler) — 主线程，运行关键监控任务
  - `nonblockSched` (BackgroundScheduler) — 后台线程，运行非关键任务
- **共享内存控制：** `/var/daemon_info` (mmap文件)
  - 字节 `G` = 正常运行模式（监控任务激活）
  - 字节非 `G` = 维护模式（监控任务暂停）
- **自检机制：** 每60秒检查心跳，如果僵死超过60秒 → 启动 guard.py + `killall -9 daemon.py`

#### guardProc.py (通用进程守护)
- **作用：** 管理 `daemon.d/*.run` 文件定义的所有进程
- **监控间隔：** 3秒
- **支持热加载：** 监控 daemon.d 目录 mtime 变化，自动重新加载进程列表
- **特别监控：** nginx、php-fpm、redis、sshd、log_agent、杀毒引擎等有独立监控逻辑

#### guardClass.py (检测引擎守护)
- **作用：** 管理 class (检测引擎) 进程
- **监控间隔：** 3秒
- **启动条件：** 对应 server 进程存活 ≥ 15秒
- **杀死条件：** 对应 server 进程死亡、实例数超限、单实例内存 >2GB 或总内存 >90%
- **启动参数：** `nice -n -10`, NUMA绑定, OOM score=15

#### guardServer.py (数通引擎守护)
- **作用：** 管理 server (数通引擎) 和 swbypass (软件bypass) 进程
- **启动顺序：** swbypass 必须先运行 → server 才能启动
- **重启前清理：** 杀 NsTlsProxy → 杀 ns_anti_virus → 杀所有 class → 杀 server
- **启动参数：** `nice -n -20`, NUMA绑定

### 3.2 Python 脚本日志体系

所有 Python 脚本使用统一的 `box.log` 日志模块：

```
日志路径规则: /var/log/<sys.argv[0]路径>.log
轮转规则: 2MB 轮转，保留1个备份
```

**关键日志文件：**

| 进程 | 日志路径 |
|------|----------|
| daemon.py | `/var/log/opt/nsfocus/bin/daemon/daemon.py.log` |
| guard.py | `/var/log/opt/nsfocus/bin/guard.py.log` |
| fsm.py | `/var/log/opt/nsfocus/bin/fsm/fsm.py.log` |
| oam.py | `/var/log/opt/nsfocus/bin/oam/oam.py.log` |

**审计日志：** 写入 PostgreSQL `audit` 表，同时通过 UDP:50002 发送给 webtoid

### 3.3 Python 层常见问题排查

#### 问题现象1: 某进程频繁重启

**排查步骤：**
```bash
# 1. 查看 daemon 日志，定位哪个进程被重启
tail -200 /var/log/opt/nsfocus/bin/daemon/daemon.py.log | grep -E "restart|start|kill"

# 2. 查看对应进程的运行定义
cat /opt/nsfocus/bin/daemon.d/<进程名>.run

# 3. 手动运行查看报错
# 以 guard.py 为例：
nice -n 0 /opt/nsfocus/bin/guard.py &
# 观察输出

# 4. 检查 daemon.d 目录是否有异常
ls -la /opt/nsfocus/bin/daemon.d/
```

#### 问题现象2: daemon.py 自身不断重启

**排查步骤：**
```bash
# 1. 查看 daemon 自身日志
tail -500 /var/log/opt/nsfocus/bin/daemon/daemon.py.log

# 2. 检查共享内存状态
xxd /var/daemon_info | head -1
# 正常应显示 "GG"

# 3. 检查 APScheduler 调度日志
tail -100 /tmp/blockSched.log

# 4. 检查是否有 guard.py 在反复拉起

# 5. 手动前台运行 daemon 查看错误
python /opt/nsfocus/bin/daemon/daemon.py
```

#### 问题现象3: webtoid 事件丢失/积压

**排查步骤：**
```bash
# 1. 检查 webtoid 进程
ps aux | grep webtoid

# 2. 查看 webtoid 日志
tail -200 /var/log/opt/nsfocus/bin/webtoid/webtoid_security.py.log

# 3. 检查 webtoid 端口监听
netstat -ntulp | grep 50002

# 4. 检查消息队列大小 (/tmp/ 下相关文件)
ls -la /tmp/ | grep -i toid

# 5. 检查事件配置

```

#### 问题现象4: ZMQ 服务调用失败

**排查步骤：**
```bash
# 1. 确认 ZMQ 端口监听
netstat -ntlp | grep -E '62000|62010|62015|62020|62025|62030|62035|62050'

# 例如 62000 = daemon
ps aux | grep daemon.py

# 3. 查看 service.json 确认服务映射
cat /opt/nsfocus/etc/service.json

# 4. 检查 ZMQ 通信日志
tail -200 /var/log/opt/nsfocus/bin/daemon/daemon.py.log | grep -i zmq
```

## 4. Server 数通引擎排查

### 4.1 数通引擎工作原理

```
                    业务网卡 (DPDK接管)
                          │
                          ▼
              ┌─────────────────────┐
              │   DPDK 收包驱动      │  (igb_uio / vfio-pci)
              └─────────┬───────────┘
                        │
              ┌─────────▼───────────┐
              │   do_netio()        │  每个 lcore 独立循环
              │   - 批量收包         │
              │   - 接口匹配         │
              │   - 预处理/Hook      │
              └─────────┬───────────┘
                        │
              ┌─────────▼───────────┐
              │   Ring Buffer       │  DPDK 共享内存 (triring)
              │   (Server → Class)  │  主环 + 高级环 × 3
              └─────────┬───────────┘
                        │
              ┌─────────▼───────────┐
              │   Class 检测引擎     │  消费数据包
              └─────────────────────┘

并行线程:
  ├─ netio_config:    配置轮询 (每1秒: doApplyConfig + doSigProcess)
  ├─ nsconfig_link_sync: 网卡链路状态监控 (inotify)
  └─ check_heart_beat: 检测引擎心跳监控 (每1秒)
```

**初始化顺序（关键！）：**
1. `ignore_signals()` — 忽略信号38
2. `Getargs().init()` — 解析启动参数
3. `addself_global_cgroup()` — 资源隔离
4. `check_base_struct_size()` — 结构体对齐校验
5. `NetioSetGlobalVars()` — 全局变量
6. `init_zmq()` — ZMQ初始化
7. `initialize_dpdk()`:
   - CPU拓扑 → 时间 → DPDK定时器 → 网卡 → Server队列 → Client队列 → 抓包引擎 → SFFLite → Hook → 信号
8. 启动线程: netio_config / check_heart_beat / nsconfig_link_sync
9. `rte_eal_mp_remote_launch(netio_launch)` — 启动所有 lcore 工作循环

### 4.2 数通引擎状态文件

| 文件 | 用途 |
|------|------|
| `/tmp/server_stat` | 引擎心跳文件 (多NUMA: `_N0`, `_N1`) |
| `/tmp/server_stat_N0` | NUMA Node 0 引擎心跳 |
| `/tmp/server_stat_N1` | NUMA Node 1 引擎心跳 |
| `/tmp/interface_pkt_stat` | 接口报文统计 |
| `/tmp/flowinfo` | 流量统计信息 |
| `/tmp/mempool.txt` | DPDK 内存池信息 |
| `/tmp/net/linkstatus/*` | 接口链路状态 (inotify监控) |
| `/tmp/net/` | 网络状态临时目录 |

### 4.3 数通引擎常见问题排查

#### 问题现象1: 引擎不启动 (进程不存在)

**排查步骤：**
```bash
# 1. 检查 server 进程是否存在
ps aux | grep -E 'server|mp_client'

# 2. 检查 swbypass 是否运行 (先决条件)
ps aux | grep swbypass

# 3. 查看 daemon 日志中的 server 启动信息
grep -i "server" /var/log/opt/nsfocus/bin/daemon/daemon.py.log | tail -50

# 4. 检查二进制文件是否存在
ls -la /opt/nsfocus/bin/server

# 5. 检查 XML 配置是否损坏
xmllint --noout /opt/nsfocus/etc/configure.xml
xmllint --noout /opt/nsfocus/etc/class.conf

# 6. 检查共享内存控制标记
xxd /var/daemon_info | head -1
# 必须显示 "GG" 才表示系统处于正常运行模式

# 7. 检查 DPDK 环境
cat /proc/meminfo | grep Huge
ls /mnt/huge/
```

#### 问题现象2: 引擎频繁重启

**排查步骤：**
```bash
# 1. 查看引擎心跳状态
cat /tmp/server_stat
# 正常应包含 "class... alive: 1"

# 2. 检查心跳是否在更新
watch -n 1 'cat /tmp/server_stat'

# 3. 查看 coredump
ls -la /tmp/coredump/
cat /tmp/coredump/CoredumpLog.txt | tail -100

# 4. 检查 DPDK 内存池信息
cat /tmp/mempool.txt

# 5. 检查 OOM 情况
dmesg | grep -i "out of memory\|oom"
grep -i "oom" /var/log/messages | tail -20

# 6. 查看 daemon 中 server 守护日志
grep "start_server\|stop_engine\|swbypass" /var/log/opt/nsfocus/bin/daemon/daemon.py.log | tail -100

# 7. 检查网卡驱动状态
cat /proc/net/packet
```

#### 问题现象3: 收包异常 / 流量不通

**排查步骤：**
```bash
# 1. 检查网卡链路状态
cat /tmp/net/linkstatus/*

# 2. 检查 DPDK 绑定的网卡
```

## 5. Class 检测引擎排查

### 5.1 检测引擎工作原理

```
DPDK Ring Buffer 收包
        │
        ▼
rte_class_rx_burst() (三环缓冲批量收包)
        │
        ▼
handle_packet() 主循环 (每个数据包)
  ├─ 1. 接口查找 (interface lookup)
  ├─ 2. IP分片重组 (IP defrag)
  ├─ 3. 协议解码 (HTTP/DNS/SMTP/FTP/SSH/... 95+解码器)
  ├─ 4. 流表查找/创建 (TCP session tracking)
  ├─ 5. 防火墙规则匹配 (stateful firewall, 86+ 文件)
  ├─ 6. IPS 规则匹配 (Hyperscan正则引擎)
  ├─ 7. Zealot IPS 引擎 (SQL注入/弱密码/XSS/文件还原)
  ├─ 8. 应用识别 (URL分类/DNS安全/SSL解密)
  ├─ 9. 威胁情报匹配 (NTI/CC通信/僵尸网络)
  ├─10. DLP 敏感数据检测
  ├─11. 反病毒扫描 (AV)
  ├─12. DDoS 检测 (应用层/CC)
  ├─13. 告警生成 (alert aggregation/merge)
  └─14. 结果写回 Ring Buffer → log_agent → 存储
        │
        ▼
每 ~1秒: every_cycle() 周期任务
  - 统计上报
  - 告警合并
  - 老旧流清理
  - 配置检查
```

**后台线程：**
| 线程 | 功能 | 间隔 |
|------|------|------|
| Statistic | 系统状态统计上报 | 4秒/10秒/30秒 |
| doConfig | ZMQ配置订阅 + 心跳监控 | 实时 |
| check_tc_malloc | tcmalloc 内存回收 | 实时 |

### 5.2 检测引擎状态文件

| 文件 | 用途 |
|------|------|
| `/tmp/cla.out.<id>` | 每个 class 实例的标准输出/错误 |
| `/tmp/fw_rule/class_finished_num` | 配置加载完成计数器 |
| `/tmp/fw_rule/class_result` | 所有实例加载完成的标记 |
| `/var/run/flock.class.n.<node>` | 每节点单实例文件锁 |
| `/opt/nsfocus/log/class_stuck_<timestamp>_<rank>` | 僵死时导出的当前数据包 |

### 5.3 检测引擎信号处理

| 信号 | 编号 | 触发行为 |

| 35 | SIGRTMIN+1 | 报文过滤重载 |
| 37 | SIGRTMIN+3 | NSHA 状态重载 |
| 38 | SIGRTMIN+4 | 时间同步 |
| 39 | SIGRTMIN+5 | DNS 刷新 |
| 40 | SIGRTMIN+6 | Server2Ext 启动 |
| 41 | SIGRTMIN+7 | Server2Ext 停止 |
| 42 | SIGRTMIN+8 | 资产识别 |
| 44 | SIGRTMIN+10 | 资产清除 / 引擎停止 |

## 8. 常用排查命令

## 附录A: 进程启动参数参考

### Server (数通引擎)
```bash
nice -n -20 ./mp_client -c <CPUMASK_HEX> -n <NODE_ID> >/dev/null 2>&1 &
# 或
nice -n -20 ./server [参数] &
# DPDK=1 环境变量
```

### Class (检测引擎)
```bash
nice -n -10 RANK=<RANK_ID> CPUMASK=<MASK_HEX> DPDK=1 ./cla 1 2 >/tmp/cla.out.<RANK_ID> 2>&1 &
```

### 返回值含义
- Server/Class 返回 0: 正常退出
- Server/Class 返回 非0: 初始化失败
- Class `CreateEngine()` 失败: 会调用 `recovery_config()` 恢复原始配置后 exit(0)

## 附录B: 故障严重级别定义

| 级别 | 现象 | 建议处理 |
|------|------|----------|
| **P0 紧急** | 设备完全不可达/流量全断 | 立即检查物理链路、电源、bypass 状态 |
| **P1 严重** | 检测引擎不工作/大量告警丢失 | 检查 engine 状态、配置加载、资源使用 |
| **P2 重要** | Web界面异常/部分功能不可用 | 检查 Web 层服务、数据库、ZMQ 通信 |
| **P3 一般** | 性能下降/偶发告警/日志异常 | 检查资源、阈值配置、日志轮转 |
| **P4 提示** | 配置告警/预判风险 | 例行检查磁盘、授权到期、证书过期 |
